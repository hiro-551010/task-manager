# フロントエンドアーキテクチャ原則

フレームワーク・言語に依存しない原則。プロジェクト固有の実装詳細は `docs/30_frontend/architecture.md` を参照すること。

---

## 基本思想：フロントエンドはプレゼンテーション層である

バックエンドの DDD においてプレゼンテーション層がアプリケーション層を呼び出すだけであるように、フロントエンドも同じ立ち位置にある。

フロントエンドの責務は **「ユーザーとバックエンドの間を仲介すること」** に限定する。

**フロントエンドがやること：**
- ユーザーの操作を受け取り、バックエンドへのリクエストに変換する
- バックエンドからのレスポンスを受け取り、ユーザーに表示する
- UI 状態（ローディング・エラー・楽観的更新）を管理する
- 入力の形式バリデーション（バックエンドと共有するスキーマを使う）

**フロントエンドがやってはいけないこと：**
- ビジネスルールの実装（「タスクは todo からしか in_progress にできない」等）
- バックエンドにある計算・判定の再実装
- API コントラクト外のバックエンド内部構造への依存

---

## レイヤー構成

```
┌──────────────────────────────────────────┐
│          ルーティング層                    │  URL ↔ 画面のマッピング・初期データ取得
├──────────────────────────────────────────┤
│          アプリケーション層                │  ミューテーション操作のオーケストレーション
├──────────────────────────────────────────┤
│          API クライアント層               │  HTTP 通信・レスポンスの型付け
├──────────────────────────────────────────┤
│          UI コンポーネント層              │  純粋な描画・ユーザー操作のキャプチャ
└──────────────────────────────────────────┘
              ↕ API コントラクト（共有型・スキーマ）
┌──────────────────────────────────────────┐
│          バックエンド（DDD 層）            │
└──────────────────────────────────────────┘
```

---

## 各層の責務

### ルーティング層

- URL と画面のマッピングを担う
- 初期表示に必要なデータを API クライアント層経由で取得し、UI コンポーネント層に渡す
- ビジネスロジックを持たない
- UI の描画責務を持たない（コンポーネントに委譲する）

### アプリケーション層（ミューテーション）

- フォーム送信・ボタン操作など、ユーザーが起こした「変更操作」を処理する
- 共有スキーマによる入力の形式バリデーションを行う（ビジネスルールの検証はしない）
- API クライアント層を呼び出す
- 操作後のUI副作用（キャッシュ更新・画面遷移・通知）を処理する
- UI コンポーネントから呼び出されるが、UI の描画には関与しない

### API クライアント層

- HTTP 通信のラッパー。フロントエンド全体でここだけが HTTP を知っている
- バックエンドとの共有型でレスポンスを型付けする
- ネットワークエラー・API エラーをここで統一的に処理する
- UI コンポーネントから直接呼ばれない（ルーティング層・アプリケーション層経由）

### UI コンポーネント層

- データを受け取って描画するだけ。受け取ったデータを加工しない
- API を直接呼ばない。操作はコールバック（イベント）として上位層に委譲する
- **bounded-context 単位でコンポーネントを整理する**（バックエンドの `modules/<ctx>/` と対応）
- UI 状態（展開・折りたたみ・フォーム入力値など）のみを自身で管理する

---

## 原則一覧

| # | 原則 |
|---|---|
| 1 | UI コンポーネントは API を直接呼ばない |
| 2 | ビジネスルールの検証はバックエンドに任せる。フロントは形式・型の検証のみ |
| 3 | bounded-context の境界を UI でも守る（`features/<ctx>/` 単位で整理） |
| 4 | バックエンドとの共有型・スキーマは必ず共有パッケージを使う（フロントで再定義しない） |
| 5 | API コントラクトがフロントとバックの唯一の結合点 |
| 6 | ルーティング層はデータの取得と受け渡しのみ。UI ロジックを持たない |

---

## バックエンド DDD との対応

| バックエンド | フロントエンド |
|---|---|
| presentation 層 | ルーティング層 |
| application 層（ハンドラー） | アプリケーション層（ミューテーション） |
| infrastructure 層（Repository 実装） | API クライアント層 |
| domain 層 | **存在しない**（ドメインロジックはバックエンドにのみ置く） |
| shared_kernel | 共有パッケージ（`packages/shared` 等） |

フロントエンドに domain 層が存在しないことは欠落ではなく、**意図的な設計**である。ドメインロジックをバックエンドに集約することで、フロント・バックで同じルールが二重管理されることを防ぐ。

## 命名規則（汎用）

| 役割 | 命名パターン | 例 |
|---|---|---|
| API クライアント | `<ctx>.ts` または `<ctx>-api.ts` | `tasks.ts` / `users-api.ts` |
| ミューテーション | `<ctx>-actions.ts` | `task-actions.ts` |
| 機能コンポーネント | `<ctx>-<役割>.tsx` | `task-list.tsx` / `task-form.tsx` |
